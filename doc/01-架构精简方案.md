## 一、统一架构决策

**本方案是其余 5 个方向的基础。所有改造方案均遵循以下架构决策：**

| 决策项 | 结论 | 理由 |
|--------|------|------|
| **核心语言** | Python 优先 | 快速迭代 Agent 逻辑，丰富的 LLM/工具生态 |
| **Rust 组件** | 保留为**可选增强** | 现有的 sandbox、indexer、server 质量高，不用白不用 |
| **CLI 入口** | Python 为主 | `likecodex` 命令直接由 Python 提供，不依赖 Rust 编译 |
| **Web UI** | 内嵌到 Python 包 | 静态构建产物随 pip 包一起分发，用户无需 Node.js |
| **分发方式** | `pip install likecodex` | 单命令安装，零额外依赖 |
| **Rust 依赖** | 降级为 `[extras]` | `pip install likecodex[sandbox]` 才需要 Rust |

## 二、当前架构问题分析

### 2.1 依赖链冗余

```
用户安装流程:
  git clone → 安装 Rust → 安装 Python 3.11 → 安装 uv → 安装 Node.js 20
  → uv sync → cargo install → npm install → npm run build

运行流程（3 个进程）:
  Python 引擎 (uv run python -m likecodex_engine.server :9090)
    → Rust 服务器 (likecodex-server :8080，薄代理层)
      → Web UI (next dev :3000)
```

### 2.2 关键数据

| 指标 | 数值 |
|------|------|
| 硬依赖数 | 4（Rust + Python + Node.js + uv） |
| 运行进程数 | 3（engine + server + web） |
| 用户从零到可用步骤 | ≈12 步 |
| 首次启动耗时 | 5-15 分钟（编译 Rust + 下载依赖） |
| Rust 服务器代码行数 | ≈800 行（核心逻辑，不含 CLI） |
| Rust 服务器功能 | 40 个路由中 37 个是透传到 Python 引擎 |

### 2.3 Rust 服务器冗余分析

[engine_bridge.rs](file:///d:/App/AgentProjects/likecodex/likecodex/crates/likecodex-server/src/engine_bridge.rs) 的核心职责是 HTTP 透传。CLI 完全可以绕过它直接访问 Python 引擎：

```
当前: CLI → Rust(:8080) → Python(:9090)
优化: CLI → Python(:9090)
```

## 三、改造方案

### 3.1 Phase 1：Python CLI 入口（1 周）

在 `likecodex_engine/` 下创建 CLI 入口，Python 直接提供 `likecodex` 命令。

**新增文件**: `likecodex_engine/cli.py`

```python
"""LikeCodex Python CLI - 主入口"""
import argparse
import asyncio
from likecodex_engine.server import start_server
from likecodex_engine.agent.loop import AgentLoop

def main():
    parser = argparse.ArgumentParser(prog="likecodex")
    parser.add_argument("prompt", nargs="?", help="一次性执行提示词")
    parser.add_argument("--chat", action="store_true", help="交互式聊天模式")
    parser.add_argument("--web", action="store_true", help="启动 Web UI")
    parser.add_argument("--port", type=int, default=9090, help="引擎端口")
    parser.add_argument("--direct", action="store_true", 
                        help="直连引擎模式（跳过 Rust 服务器）")
    # ... 更多参数
    
    args = parser.parse_args()
    
    if args.web:
        # 启动引擎 + 内嵌 Web UI
        asyncio.run(start_with_web(args.port))
    elif args.chat or not args.prompt:
        # 交互模式
        asyncio.run(interactive_chat())
    else:
        # 一次性执行
        asyncio.run(run_once(args.prompt))
```

**修改文件**: `pyproject.toml`（根目录）

```toml
[project.scripts]
likecodex = "likecodex_engine.cli:main"
```

这样用户 `pip install likecodex` 后直接 `likecodex "帮我重构这个函数"` 就能用。

**保留兼容性**: Rust CLI 的 `likecodex` 仍然存在，但升级检查逻辑：

```rust
// crates/likecodex-cli/src/main.rs
// 检测是否有 Python 版 CLI，优先使用 Python 版（更快启动）
if !args.force_rust && has_python_cli() {
    exec_python_cli(args)
}
```

### 3.2 Phase 2：Rust 组件可选化（2 周）

将 Rust crate 标记为可选组件，通过 extras 机制管理。

**修改文件**: `pyproject.toml`

```toml
[project.optional-dependencies]
sandbox = []    # 需要 Rust likecodex-sandbox
indexer = []    # 需要 Rust likecodex-indexer
full = ["likecodex[sandbox,indexer]"]  # 完整安装
```

**修改文件**: `likecodex_engine/config_loader.py`

```python
class ComponentAvailability:
    """运行时检测可选组件"""
    sandbox_available: bool = False
    indexer_available: bool = False
    
    @classmethod
    def detect(cls):
        """检测已安装的可选组件"""
        cls.sandbox_available = shutil.which("likecodex-sandbox") is not None
        cls.indexer_available = shutil.which("likecodex-indexer") is not None
        # 如果没有 Rust 组件，使用 Python 回退实现
```

### 3.3 Phase 3：Rust 组件功能 Python 化（可选，长期）

对于高频功能，在 Python 中提供轻量回退实现，使 Rust 成为纯性能优化而非功能依赖：

| Rust 组件 | 当前功能 | Python 回退方案 |
|-----------|---------|----------------|
| `likecodex-sandbox` | Docker 沙箱 | subprocess 直接调用 docker CLI（已部分实现） |
| `likecodex-indexer` | 文件索引 + CodeGraph | 使用 `tree_sitter` Python 绑定 或 `pywalk` |
| `likecodex-server` | HTTP 透传 + 事件聚合 | aiohttp 服务器已提供（[server.py](file:///d:/App/AgentProjects/likecodex/likecodex/packages/likecodex-engine/likecodex_engine/server.py)） |
| `likecodex-executor` | 本地命令执行 | `asyncio.create_subprocess_exec` 即可，无需 Rust |

## 四、改造后的架构

### 4.1 目标架构

```
轻量模式（默认）:
  pip install likecodex
  likecodex --chat
  └── Python 进程 (aiohttp :9090 + AgentLoop + CLI)

完整模式:
  pip install likecodex[full]
  likecodex --web
  └── Python 进程 (:9090 + AgentLoop + 静态 Web UI)
  └── Docker 沙箱 (可选)
  └── Rust indexer (可选，性能优化)
```

### 4.2 安装流程对比

| 环节 | 改造前 | 改造后 |
|------|--------|--------|
| 最小依赖 | Rust + Python + Node.js + uv | Python >=3.11 + pip |
| 安装命令 | 多步脚本 | `pip install likecodex` |
| 安装耗时 | 5-15 分钟 | 30 秒 |
| 启动命令 | `likecodex start --web` | `likecodex --web` |
| 启动耗时 | 3-5 秒（含子进程启动） | <1 秒（单进程） |

## 五、风险与注意事项

| 风险 | 概率 | 缓解措施 |
|------|------|----------|
| Python 启动比 Rust 慢 | 中 | 引入缓存预热机制（见方向三） |
| 现有 Rust CLI 用户升级冲突 | 低 | 保留 `likecodex --force-rust` 兼容 |
| Python CLIno 丢失 TUI 功能 | 中 | TUI 用 `textual` 或 `rich` 替代 ratatui |
| Rust sandbox 安全性更好 | 中 | Python 版 sandbox 同样调用 docker CLI |
